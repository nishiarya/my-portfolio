üéØ React Native Fundamentals

What is React Native and how does it work?

React Native is a framework for building native mobile applications using JavaScript and React. It works by:

  • JavaScript Thread: Runs your business logic and React code
  • Native Thread: Handles UI rendering with native components
  • Bridge (Old Architecture): Serializes data between JS and Native threads using JSON
  • JSI (New Architecture): Direct communication without serialization
üí° Interview Tip
Always mention both old (Bridge) and new (JSI) architecture to show you're up-to-date.
What's the difference between React Native and other cross-platform frameworks?
Framework Rendering Performance Code Sharing
React Native Native Components Near-Native ~90%
Flutter Custom Rendering (Skia) Near-Native ~95%
Ionic/Cordova WebView Lower ~99%
Native (Swift/Kotlin) Native Best 0%

RN Advantage: Uses actual native UI components, large ecosystem (React/JS), hot reloading, strong community.

Explain the Virtual DOM in React Native context

React Native uses a Virtual DOM (VDOM) similar to React web:

  1. Initial Render: Creates VDOM tree from your components
  2. State Change: Creates new VDOM tree
  3. Diffing: Compares old vs new VDOM (Reconciliation)
  4. Batch Updates: Calculates minimal changes needed
  5. Native Update: Sends only changed parts to native side

What is Props Drilling and how to avoid it?

Props Drilling: Passing props through multiple component levels that don't need them.

// ‚ùå Props Drilling Problem <App user={user}> <Layout user={user}> // doesn't use user <Sidebar user={user}> // doesn't use user <UserAvatar user={user} /> // actually uses user </Sidebar> </Layout> </App>

Solutions:

  1. React Context: For app-wide state (theme, auth, locale)
  2. Redux/Zustand: For complex global state
  3. Component Composition: Pass components as children
// ‚úÖ Solution: Context API const UserContext = createContext(); const App = () => ( <UserContext.Provider value={user}> <Layout> <Sidebar> <UserAvatar /> // uses useContext(UserContext) </Sidebar> </Layout> </UserContext.Provider> );
Explain React.memo() and when to use it

React.memo() is a Higher Order Component that memoizes functional components to prevent unnecessary re-renders.

import React, { memo } from 'react'; // Without memo - re-renders on every parent render const ExpensiveComponent = ({ data }) => { // expensive rendering logic return <View>...</View>; }; // With memo - only re-renders if props change const ExpensiveComponent = memo(({ data }) => { return <View>...</View>; }); // With custom comparison const ExpensiveComponent = memo( ({ data }) => <View>...</View>, (prevProps, nextProps) => prevProps.data.id === nextProps.data.id );
‚ö†Ô∏è Don't Overuse
memo() has overhead. Use only for components with expensive renders or frequent parent updates.

What is Zustand and when would you choose it over Redux?

Zustand is a minimal state management library with a simple API:

  • Don't need Redux DevTools extensively
  • Building small to medium apps
  • Want fast learning curve for team

Explain useCallback and useMemo with examples

useCallback: Memoizes a function reference
useMemo: Memoizes a computed value

import { useCallback, useMemo, useState } from 'react'; const ProductList = ({ products, category }) => { const [sortOrder, setSortOrder] = useState('asc'); // ‚úÖ useMemo - for expensive calculations const filteredProducts = useMemo(() => { return products .filter(p => p.category === category) .sort((a, b) => sortOrder === 'asc' ? a.price - b.price : b.price - a.price ); }, [products, category, sortOrder]); // Only recalculates when these change // ‚úÖ useCallback - for function references (passed to children) const handleProductPress = useCallback((productId) => { navigation.navigate('ProductDetail', { productId }); }, [navigation]); return ( <FlatList data={filteredProducts} renderItem={({ item }) => ( <ProductCard product={item} onPress={handleProductPress} // Stable reference /> )} /> ); }; // ProductCard uses React.memo, so stable onPress prevents re-renders const ProductCard = memo(({ product, onPress }) => { return ( <TouchableOpacity onPress={() => onPress(product.id)}> <Text>{product.name}</Text> </TouchableOpacity> ); });
‚ö†Ô∏è Common Mistake
Don't wrap everything in useCallback/useMemo. They have overhead. Use only when:
  • Passing callbacks to memoized children
  • Expensive calculations (sorting large arrays)
  • Values used in other hooks' dependency arrays
Create a custom hook for API data fetching
// hooks/useApi.ts import { useState, useEffect, useCallback } from 'react'; interface UseApiState<T> { data: T | null; loading: boolean; error: Error | null; refetch: () => void; } function useApi<T>( fetchFn: () => Promise<T>, deps: any[] = [] ): UseApiState<T> { const [data, setData] = useState<T | null>(null); const [loading, setLoading] = useState(true); const [error, setError] = useState<Error | null>(null); const fetchData = useCallback(async () => { try { setLoading(true); setError(null); const result = await fetchFn(); setData(result); } catch (err) { setError(err as Error); } finally { setLoading(false); } }, [fetchFn]); useEffect(() => { fetchData(); }, [...deps, fetchData]); return { data, loading, error, refetch: fetchData }; } // Usage const UserProfile = ({ userId }) => { const { data: user, loading, error, refetch } = useApi( () => api.getUser(userId), [userId] ); if (loading) return <Spinner />; if (error) return <ErrorView onRetry={refetch} />; return <Text>{user?.name}</Text>; };
What are the rules of hooks?
  1. Only call at top level: Don't call hooks inside loops, conditions, or nested functions
  2. Only call from React functions: Call from functional components or custom hooks
  3. Use ESLint plugin: eslint-plugin-react-hooks
// ‚ùå Wrong - conditional hook if (condition) { const [state, setState] = useState(); // Error! } // ‚úÖ Correct - hook at top level const [state, setState] = useState(); if (condition) { // use state here }

üèóÔ∏è React Native New Architecture

Explain React Native's New Architecture (Fabric & TurboModules)

The New Architecture consists of:

1. JSI (JavaScript Interface)

  • Replaces the bridge with direct C++ bindings
  • Synchronous calls between JS and Native
  • No JSON serialization overhead

2. Fabric (New Rendering System)

  • Renders UI synchronously when needed
  • Better support for concurrent features
  • Improved priority scheduling
  • Reduced memory usage

3. TurboModules (New Native Modules)

  • Lazy loading of native modules
  • Type-safe communication via CodeGen
  • Direct JSI access (no bridge)

4. CodeGen

  • Generates native code from TypeScript/Flow specs
  • Ensures type safety across JS/Native boundary
Feature Old Architecture New Architecture
Communication Bridge (JSON) JSI (Direct C++)
Native Modules All loaded at startup Lazy loaded (TurboModules)
Rendering Async only Sync + Async (Fabric)
Type Safety Runtime checks Build-time (CodeGen)
üí° Interview Must-Know
New Architecture is enabled by default in RN 0.74+. Always mention you're familiar with migrating existing apps.
How does JSI improve performance over the Bridge?

Old Bridge Architecture:

JS Thread ‚îÄ‚îÄ[serialize to JSON]‚îÄ‚îÄ> Bridge Queue ‚îÄ‚îÄ[deserialize]‚îÄ‚îÄ> Native Thread ‚Üë ‚Üì ‚îî‚îÄ‚îÄ‚îÄ‚îÄ[serialize]‚îÄ‚îÄ‚îÄ‚îÄ‚îÄ‚îÄ Queue ‚îÄ‚îÄ‚îÄ‚îÄ[deserialize]‚îÄ‚îÄ‚îÄ‚îÄ‚îò

Problems: Async only, JSON overhead, queue bottleneck

New JSI Architecture:

JS Thread ‚Üê‚îÄ‚îÄ[direct C++ calls]‚îÄ‚îÄ‚Üí Native Thread (synchronous when needed)

Benefits:

  • Synchronous calls: No waiting for bridge queue
  • No serialization: Direct memory access
  • Shared ownership: JS can hold references to native objects
  • Faster startup: Lazy module loading
// Old Bridge - Async with JSON serialization NativeModules.MyModule.getData().then(data => ...); // New JSI - Can be synchronous const data = global.myModule.getDataSync(); // Direct call!
What is Hermes and why use it?

Hermes is a JavaScript engine optimized for React Native:

  • Ahead-of-Time (AOT) Compilation: JS is compiled to bytecode at build time
  • Faster Startup: 40-50% faster TTI (Time to Interactive)
  • Lower Memory: 20-30% reduction in memory usage
  • Smaller APK: Bytecode is smaller than JavaScript
  • Optimized for Mobile: Better garbage collection for mobile constraints
// android/app/build.gradle project.ext.react = [ enableHermes: true // Enable Hermes ] // Check if Hermes is enabled at runtime const isHermes = () => !!global.HermesInternal; console.log('Using Hermes:', isHermes());
üí° Important
Hermes is now the default engine in React Native (0.70+). It's required for the New Architecture.

What causes performance issues in React Native and how to debug them?

Common Performance Issues:

  1. JS Thread Blocking: Heavy computations, large JSON parsing
  2. Excessive Re-renders: Missing memoization, unstable references
  3. Bridge Congestion: Too many cross-bridge calls
  4. Large Images: Unoptimized images, missing caching
  5. Memory Leaks: Uncleaned listeners, retained references

Debugging Tools:

  • React DevTools Profiler: Identify slow components
  • Flipper: Network, Layout, Performance inspection
  • Systrace (Android): Low-level performance traces
  • Xcode Instruments (iOS): CPU, Memory profiling
  • why-did-you-render: Detect unnecessary re-renders
// Detect unnecessary re-renders import whyDidYouRender from '@welldone-software/why-did-you-render'; whyDidYouRender(React, { trackAllPureComponents: true, }); // Add to specific component MyComponent.whyDidYouRender = true;

Explain Closures with a practical React Native example

A closure is a function that has access to variables from its outer scope, even after the outer function has returned.

// Closure example in React Native const CounterButton = () => { const [count, setCount] = useState(0); // This function "closes over" count variable const handlePress = () => { // count is accessed from outer scope console.log(`Current count: \${count}`); setCount(count + 1); }; return <Button onPress={handlePress} title={`Count: \${count}`} />; }; // ‚ö†Ô∏è Stale Closure Problem const Timer = () => { const [count, setCount] = useState(0); useEffect(() => { const interval = setInterval(() => { // ‚ùå Bug: count is always 0 (stale closure) setCount(count + 1); }, 1000); return () => clearInterval(interval); }, []); // Empty deps = closure captures initial count // ‚úÖ Fix: Use functional update useEffect(() => { const interval = setInterval(() => { setCount(prev => prev + 1); // Always gets latest }, 1000); return () => clearInterval(interval); }, []); };

Design a scalable React Native app architecture

Folder Structure:

src/ ‚îú‚îÄ‚îÄ api/ # API layer ‚îÇ ‚îú‚îÄ‚îÄ client.ts # Axios/fetch instance ‚îÇ ‚îú‚îÄ‚îÄ endpoints/ # API endpoints by feature ‚îÇ ‚îî‚îÄ‚îÄ types/ # API types ‚îú‚îÄ‚îÄ assets/ # Images, fonts ‚îú‚îÄ‚îÄ components/ # Shared components ‚îÇ ‚îú‚îÄ‚îÄ ui/ # Design system (Button, Input, etc.) ‚îÇ ‚îî‚îÄ‚îÄ common/ # Business components ‚îú‚îÄ‚îÄ features/ # Feature-based modules ‚îÇ ‚îú‚îÄ‚îÄ auth/ ‚îÇ ‚îÇ ‚îú‚îÄ‚îÄ screens/ ‚îÇ ‚îÇ ‚îú‚îÄ‚îÄ components/ ‚îÇ ‚îÇ ‚îú‚îÄ‚îÄ hooks/ ‚îÇ ‚îÇ ‚îú‚îÄ‚îÄ store/ # Feature-specific state ‚îÇ ‚îÇ ‚îî‚îÄ‚îÄ types/ ‚îÇ ‚îú‚îÄ‚îÄ home/ ‚îÇ ‚îî‚îÄ‚îÄ profile/ ‚îú‚îÄ‚îÄ hooks/ # Shared custom hooks ‚îú‚îÄ‚îÄ navigation/ # Navigation setup ‚îú‚îÄ‚îÄ store/ # Global state (Redux/Zustand) ‚îú‚îÄ‚îÄ theme/ # Design tokens, colors, spacing ‚îú‚îÄ‚îÄ utils/ # Helper functions ‚îî‚îÄ‚îÄ App.tsx

Key Architecture Principles:

  1. Feature-based organization: Group by feature, not type
  2. Separation of concerns: UI, Business Logic, Data layers
  3. Dependency injection: Services, API clients
  4. Error boundaries: Graceful error handling
  5. Offline-first: Local storage, sync queues

  1. Local Database: SQLite (via WatermelonDB) or Realm
  2. Sync Queue: Queue mutations when offline
  3. Conflict Resolution: Last-write-wins or merge strategies
  4. Network Detection: NetInfo for connectivity

üë• Leadership Scenarios

How would you lead a migration to React Native's New Architecture?

Phase 1: Assessment

  • Audit current native modules and third-party libraries
  • Identify incompatible dependencies
  • Estimate effort and create timeline

Phase 2: Preparation

  • Update React Native to latest version
  • Enable Hermes engine
  • Update/replace incompatible libraries
  • Set up feature flags for gradual rollout

Phase 3: Migration

  • Convert native modules to TurboModules
  • Update custom native components for Fabric
  • Run comprehensive testing

Phase 4: Rollout

  • Internal testing with feature flag
  • Beta release to subset of users
